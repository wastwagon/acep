P3009: failed migration 20260426110650_add_portal_user_contributions
===================================================================

Symptoms: logs show "migrate found failed migrations" and the app restarts
in a loop. Prisma will not run later migrations until this is fixed.

A) Coolify without host SSH: on the **web** service, add an environment
   variable (then Redeploy):

   PRISMA_RESOLVE_ROLLED_BACK_MIGRATION=20260426110650_add_portal_user_contributions

   The entrypoint runs `migrate resolve --rolled-back` before `migrate deploy`.
   When the app is healthy, **remove** this variable and redeploy (only needed
   once; requires a build that includes the updated docker/entrypoint-web.sh).

If Postgres shows "could not receive data from client: Connection reset by
peer" while the migration file appears in the log, the *database* did not
necessarily reject the SQL — the Prisma client in the *web* container (or
whatever ran migrate) dropped the connection: container restarts, deploy
stopping the service, or OOM during migrate. The migration can still be marked
failed in _prisma_migrations (P3009). The transaction often rolls back; use
the resolve + deploy steps below. Re-run migrate from a stable one-off
(`docker run` / Coolify one-shot with DATABASE_URL) if the web app keeps
restarting and tears down the connection again.

0) If web restarts too fast to docker exec, use a one-off (host or new container)
   with the same DATABASE_URL. See the repo script:

   scripts/resolve-failed-migration-20260426110650.sh

   In a deployed web image it is copied to
   /app/scripts/resolve-failed-migration-20260426110650.sh
   (run with: sh /app/scripts/...).

   On the **Coolify host over SSH** (not inside the backend/web container: those
   images have no `docker` binary, so you will get "docker: not found" in
   Coolify’s in-browser Terminal for a service).

   With the same compose project name and .env as the app (see deployment log
   for -p / --project-name):

   docker compose -p YOUR_PROJECT_NAME -f docker-compose.coolify.yml --env-file .env --profile tools run --rm migrate-resolve

1) One-off in a shell with the SAME DATABASE_URL as the web app (e.g. Coolify
   terminal, or: docker exec -it <web-container> sh, cd /app):

   node node_modules/prisma/build/index.js migrate resolve --rolled-back 20260426110650_add_portal_user_contributions
   node node_modules/prisma/build/index.js migrate deploy

   A single failed migration in Postgres usually runs inside a transaction, so
   the database may still be in a good state. Marking the migration "rolled
   back" and re-applying is often enough.

2) If migrate deploy errors with "relation ... already exists" (partial
   state), run the SQL in:

   prisma/ops/rollback-partial-20260426110650_add_portal_user_contributions.sql

   against the acep database (e.g. psql), then:

   node node_modules/prisma/build/index.js migrate resolve --rolled-back 20260426110650_add_portal_user_contributions
   node node_modules/prisma/build/index.js migrate deploy

3) If you already applied objects manually and the schema matches, you can
   mark as applied instead of rolled back:

   node node_modules/prisma/build/index.js migrate resolve --applied 20260426110650_add_portal_user_contributions
   node node_modules/prisma/build/index.js migrate deploy

Reference: https://pris.ly/d/migrate-resolve
